00152 /**********************************************************************
00153             LP Suspension Design Notes! (John Jenkins, ANL)
00154 
00155 Many times, when developing optimistic models, we are able to
00156 determine < LP state, event > pairs which represent infeasible model
00157 behavior. These types of simulation states typically arise when time
00158 warp causes us to receive and potentially process messages in an order
00159 we don't expect.
00160 
00161 For example, consider a client/server protocol in which a server sends
00162 an ACK to a client upon completion of some event. In optimistic mode,
00163 the client can see what amounts to duplicate ACKs from the server due
00164 to the server LP rolling back and re-sending an ACK.
00165 
00166 While some models can gracefully cope with such issues, more complex
00167 models can have troubles (the client in the example could for instance
00168 destroy the request metadata after receiving an ACK).
00169 
00170 A solution, as noted in the "Dark Side of Risk" paper, is to introduce
00171 LP "self-suspend" functionality. If an LP is able to detect a < state,
00172 message > pair which is incorrect / unexpected in a well-behaved
00173 simulation, the LP should be able to put itself into suspend mode,
00174 refusing to process messages until rolled back to a pre < state,
00175 message > state. There are two benefits: 1) it greatly reduces the
00176 difficulty in tracking down and distinguishing proper model bugs from
00177 bugs arising from time-warp related issues such as out-of-order event
00178 receipt and 2) it improves simulation performance by pruning the
00179 number of processed events that we know are invalid and will be rolled
00180 back anyways.
00181 
00182 I suggest the function signature tw_suspend(tw_lp *lp, int
00183 do_suspend_event_rc, const char * format, ...), with the following
00184 semantics:
00185 
00186 After a call to tw_suspend, all subsequent events (both forward and
00187 reverse) that arrive at the suspended LP shall be processed as if they
00188 were no-ops. The reverse event handler of the event that caused the
00189 suspend will be run if do_orig_event_rc is nonzero; otherwise, the
00190 reverse event handler shall additionally be a no-op. Typically,
00191 do_orig_event_rc == 0 is desired, as good coding practices for
00192 moderate-or-greater complexity simulations dictate state/event
00193 validation prior to modifying LP state (partial rollbacks are very
00194 undesirable), but there may be messy logic in the user code for which
00195 a partial rollback is warranted (operations that free memory as a side
00196 effect of operations, for example).  An LP exits suspend state upon
00197 rolling back the event that caused the suspend (whether or not that
00198 event is processed as a no-op).  Upon GVT, if an LP is in self-suspend
00199 mode and the event that caused the suspend has a timestamp less than
00200 that of GVT, then the simulator shall report the format string of
00201 suspended LP(s) and exit.  A NULL format string is acceptable for
00202 performance purposes, e.g. when doing "production" simulation runs.
00203 
00204 *************************************************************************/
