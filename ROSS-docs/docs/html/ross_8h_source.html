00004 /** @mainpage Rensselaer's Optimistic Simulation System (ROSS)
00005     @section intro_sec Introduction
00006     
00007     ROSS is an acronym for Rensselaer's Optimistic Simulation System. It is a
00008     parallel discrete-event simulator that executes on shared-memory
00009     multiprocessor systems. ROSS is geared for running large-scale simulation
00010     models (i.e., 100K to even 1 million object models).  The synchronization
00011     mechanism is based on Time Warp. Time Warp is an optimistic
00012     synchronization mechanism develop by Jefferson and Sowizral [10, 11] used
00013     in the parallelization of discrete-event simulation. The distributed
00014     simulator consists of a collection of logical processes or LPs, each
00015     modeling a distinct component of the system being modeled, e.g., a server
00016     in a queuing network. LPs communicate by exchanging timestamped event
00017     messages, e.g., denoting the arrival of a new job at that server.
00018 
00019     The Time Warp mechanism uses a detection-and-recovery protocol to
00020     synchronize the computation. Any time an LP determines that it has
00021     processed events out of timestamp order, it "rolls back" those events, and
00022     re-executes them. For a detailed discussion of Time Warp as well as other
00023     parallel simulation protocols we refer the reader to [8]
00024 
00025     ROSS was modeled after a Time Warp simulator called GTW or Georgia Tech
00026     Time Warp[7]. ROSS helped to demonstrate that Time Warp simulators can be
00027     run efficiently both in terms of speed and memory usage relative to a
00028     high-performance sequential simulator.
00029 
00030     To achieve high parallel performance, ROSS uses a technique call Reverse
00031     Computation. Here, the roll back mechanism in the optimistic simulator is
00032     realized not by classic state-saving, but by literally allowing to the
00033     greatest possible extent events to be reverse. Thus, as models are
00034     developed for parallel execution, both the forward and reverse execution
00035     code must be written. Currently, both are done by hand. We are
00036     investigating automatic methods that are able to generate the reverse
00037     execution code using only the forward execution code as input. For more
00038     information on ROSS and Reverse Computation we refer the interested reader
00039     to [4, 5]. Both of these text are provided as additional reading in the
00040     ROSS distribution.
